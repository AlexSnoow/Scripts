# Specification — Что и почему

Этот файл описывает доменную логику, бизнес-инварианты и требования — **что** делает проект и **почему** именно так.
Технические детали реализации см. в docs/plan.md.

## 1. Доменные инварианты файловых операций

### 1.1. Порядок операций (КРИТИЧНО)
**НИКОГДА не нарушать:**
1. Результат создаётся локально (архив, копия, синхронизация) → проверяется → затем копируется в сеть
2. **До** успешной верификации **ЗАПРЕЩЕНО** удаление источников
3. Ротация старых файлов — **после** создания новых

### 1.2. Копирование в сетевое хранилище
Копирование допускается только после:
- Проверки кода возврата архиватора (RAR: ExitCode=0, 7zip: .ExitCode=0, tar: $?=0)
- Проверки наличия файла
- Проверки размера > 0

### 1.3. Политика ротации и безопасность удаления
Удаление подчиняется правилам:
- `DaysOld` — максимальный возраст (0-3650 дней)
- `KeepCount` — минимальное количество файлов (0-100000)
- **Приоритет:** KeepCount > DaysOld

### 1.4. Полнота и трассируемость
Для каждого задания **обязательно**:
- Лог-файл: `<PCName>_<JobName>_<timestamp>.log`
- XML-отчёт: `<PCName>_<JobName>_<timestamp>.xml`
- CSV-список файлов: `<PCName>_<JobName>_<timestamp>.csv`
- Сводный отчёт: `<PCName>_summary.xml`

Для Bash: лог-файл обязателен, XML/CSV отчёты опциональны.

## 2. Логирование, ошибки и наблюдаемость

### 2.1. Обязательное логирование
Все ключевые этапы логируются (для PS — уровни INFO/WARNING/ERROR/SUCCESS/DEBUG, для Bash — префиксы [INFO]/[WARN]/[ERROR]/[OK]):
- Старт скрипта и заданий
- Основная операция (успех/ошибка)
- Верификация (результат сравнения)
- Копирование в сеть
- Ротация файлов
- Отправка почты

### 2.2. Обработка ошибок
**Обязательная конструкция (PowerShell):**
```powershell
try {
    # Критическая операция
}
catch {
    Write-Log "Ошибка: $($_.Exception.Message)" -Level ERROR
    throw
}
```

**Для Bash:**
```bash
set -euo pipefail
log "ERROR: $1"
exit 1
```

**Запрещён:** `SilentContinue`, `SilentlyContinue` для критических операций.

### 2.3. Безопасность логов
**ЗАПРЕЩЕНО** в логах:
- Пароли
- Токены
- Строки подключения
- Персональные данные

### 2.4. Уведомления
Email-уведомления:
- Сообщают агрегированный результат (Success/Error/Warning)
- Содержат ссылки на лог-файлы
- **Не дублируют** доменную логику
- Сбой отправки = `WARNING`, не `ERROR` основной операции

## 3. Конфигурация и безопасность

### 3.1. Конфигурация (единый источник правды)
Форматы:
- PS: XML ({type}-Config.xml)
- Bash: Shell conf ({type}-Config.conf, key=value)

**Структура (PS XML):**
```xml
<BackupConfig>
  <General>...</General>
  <Paths>...</Paths>
  <Recipients>...</Recipients>
  <Integrity>...</Integrity>
  <Jobs>...</Jobs>
</BackupConfig>
```

**Все пути, параметры и настройки — только в XML/conf (хардкод в скрипте запрещён).**
Хардкод значений в скрипте **запрещён** (кроме констант по умолчанию).

### 3.2. Проверка целостности (SHA256)
**Обязательные проверки перед запуском:**
1. Хеш XML-конфигурации (`$XmlHash`)
2. Хеш внешнего архиватора (RAR: RarExeHash, 7zip: SevenZipHash — в конфиге). Для Compress-Archive хеш не требуется (встроенный PS 5.1+)

**При несовпадении хеша — немедленная остановка скрипта.**

### 3.3. Внешние зависимости
**Разрешено:**
- RAR.exe, 7z.exe, tar, gzip (внешние утилиты указываются в конфиге)
- Compress-Archive (встроенный PS 5.1+, опционально — для хостов, где запрещены внешние утилиты; при отсутствии — остановка с ошибкой, см. constitution.md 2.1)

**Запрещено** без явного обоснования:
- Дополнительные конфигурационные файлы
- Сторонние модули
- Внешние API

### 3.4. Секреты и учётные данные
**ЗАПРЕЩЕНО** в скрипте и конфигурации:
- Пароли
- Токены
- Ключи API

### 3.5. Необходимые права
Права определяются для каждого задания:
- Чтение: источники данных
- Запись: назначения LocalDest, RemoteDest, LogPathRoot, NetLogPath
- Выполнение: внешний архиватор (RAR/7z/tar)
- учетная запись от имени которой выполняется скрипт, имеет право только на чтение и выполнение в каталоге запуска скрипта (для чтения файлов конфигурации и запуска скрипта)

## 4. Что проект НЕ решает

- Планирование запуска (расписание задач)
- Шифрование архивов
- Управление правами доступа на стороне серверов
- Долгосрочное хранение по регуляторным политикам
