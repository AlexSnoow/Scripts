# Конституция проекта резервного копирования PowerShell
Этот файл является проектной конституцией и хранится как docs/constitution.md.
Его положения задают неколебимые правила и ограничения, которыми обязаны руководствоваться и разработчики, и AI‑агенты при спецификации, планировании, разбиении на задачи и реализации.

## 1. Назначение и область применения
1.1. Проект реализует сценарии резервного копирования файлов и папок под Windows с использованием **единого монолитного PowerShell-скрипта** `Backup-ps2-v4.ps1`. Он обеспечивает создание локальных RAR-архивов, их последующую доставку в сетевое хранилище и управляемую ротацию старых резервных копий.

1.2. Проект **не решает** задачи планирования запуска (расписание задач), шифрования архивов, управления правами доступа на стороне серверов и долгосрочного хранения по регуляторным политикам. Эти задачи выполняются внешними системами (Task Scheduler, системы шифрования, политики хранения).

## 2. Базовые принципы разработки
### 2.1. PowerShell 2.0 и совместимость с Windows 7
**ВСЕГДА:**
- Весь код должен быть написан на PowerShell и быть **100% совместимым с PowerShell 2.0** и Windows 7.
- **Запрещено** использовать возможности PS 3.0+ без совместимых альтернатив.
- Использование внешних CLI-утилит (RAR) допускается только через изолированные функции-обёртки с чётким интерфейсом.

**Недоступные возможности PS 2.0 (использовать альтернативы):**
| Возможность                      | Альтернатива                             |
| -------------------------------- | ---------------------------------------- |
| `Get-FileHash`                   | `Get-FileHashCompat` (SHA256 через .NET) |
| `[string]::IsNullOrWhiteSpace()` | `Test-StringIsNullOrWhiteSpace`          |
| `ConvertFrom-Json`               | XML вместо JSON                          |
| `PSCustomObject`                 | `New-Object PSObject -Property @{}`      |

### 2.2. KISS и бритва Оккама
Решения должны быть максимально простыми и очевидными при сохранении надёжности и поддерживаемости.
Избыточные абстракции, «умные» трюки и чрезмерно глубокие пайплайны, усложняющие чтение и отладку, **запрещены**.

### 2.3. DRY и единый конвейер (Unified Pipeline)
Повторяющаяся логика обязана выноситься в отдельные функции; копирование кода считается архитектурной ошибкой.
**Архитектура проекта:** единый монолитный скрипт с внутренними функциями-модулями в регионах (`#region`).

### 2.4. SOLID и единственная ответственность
Каждая функция должна иметь одну чётко определённую ответственность в домене резервного копирования.
- `Start-RarArchive` — только архивация
- `Invoke-Verification` — только проверка
- `Remove-OldFiles` — только ротация

### 2.5. YAGNI и разумный BDUF
Реализовывается только функциональность, явно зафиксированная в конфигурации `Backup-Config-All.xml`; «код на будущее» без требований **запрещён**.

### 2.6. APO (Avoid Premature Optimization)
Оптимизация производительности допускается только при наличии измеримой проблемы и понятных целевых метрик.
Приоритет: корректность и простота кода важнее гипотетической оптимизации.

### 2.7. Чистый и предсказуемый код
Скрытые побочние эффекты **запрещены**: любая модификация файловой системы, конфигурации или логов должна быть очевидна из названия и описания функции.
Имена функций, параметров и файлов должны однозначно отражать назначение.

### 2.8. Comment-based help (PS 2.0)
**Каждая публичная функция обязана содержать блок help:**
```powershell
<#
.SYNOPSIS
    Краткое описание (одна строка)
.DESCRIPTION
    2-5 строк с основными сценариями
.PARAMETER Name
    Описание параметра
.EXAMPLE
    Get-Function -Param "Value"
#>
```

**Минимум:** `.SYNOPSIS`, `.DESCRIPTION`, `.PARAMETER`, `.EXAMPLE`

### 2.9. Работа с кодировками и русскими символами
**КОДИРОВКИ:**
- **Логи (OEM/CP866):** `$Script:EncodingOEM = [System.Text.Encoding]::GetEncoding(866)`
- **Отчёты (UTF8 без BOM):** `$Script:EncodingUTF8NoBOM = New-Object System.Text.UTF8Encoding $false`

Все пути, имена файлов и сообщения должны корректно обрабатывать кириллицу.

## 3. Архитектура и структура скрипта
### 3.1. Монолитная архитектура
**Текущая архитектура:** единый скрипт `Backup-ps2-v4.ps1` с функциями, организованными в регионы:
```
Backup-ps2-v4.ps1
├── Константы и настройки
├── #region ФУНКЦИЯ ВЫЧИСЛЕНИЯ ХЕША
├── #region ФУНКЦИЯ ПРОВЕРКИ ЦЕЛОСТНОСТИ
├── #region МОДУЛЬ ЛОГИРОВАНИЯ
├── #region МОДУЛЬ RAR ОПЕРАЦИЙ
├── #region МОДУЛЬ ВЕРИФИКАЦИИ
├── #region РОТАЦИЯ ФАЙЛОВ
├── #region ОТПРАВКА ПОЧТЫ
└── #region UNIFIED PIPELINE
```

### 3.2. Unified Pipeline (Единый конвейер)
**Обязательный порядок выполнения:**
1. **Подготовка элементов** — `Prepare-ArchiveItems`
2. **Архивация** — `Invoke-ArchivePipeline`
3. **Верификация** — `Invoke-Verification`
4. **Пост-операции** — `Invoke-PostOperations`

Нарушение порядка **запрещено**.

### 3.3. Режимы архивации
Скрипт поддерживает три режима, определяемые конфигурацией:
- **Normal** — весь источник в один архив
- **IndividualFiles** — каждый файл отдельный архив
- **IndividualFolders** — каждая папка отдельный архив

Добавление новых режимов — только через новые функции с сохранением интерфейса.

## 4. Доменные инварианты резервного копирования
### 4.1. Порядок операций (КРИТИЧНО)
**НИКОГДА не нарушать:**
1. Архив создаётся **локально** → проверяется → **затем** копируется в сеть
2. **До** успешной верификации **ЗАПРЕЩЕНО** удаление источников
3. Ротация старых архивов — **после** создания новых

### 4.2. Копирование в сетевое хранилище
Копирование допускается только после:
- Проверки кода возврата RAR (ExitCode = 0)
- Проверки наличия файла архива
- Проверки размера архива > 0

### 4.3. Политика ротации и безопасность удаления
Удаление подчиняется правилам:
- `DaysOld` — максимальный возраст (0-3650 дней)
- `KeepCount` — минимальное количество файлов (0-100000)
- **Приоритет:** KeepCount > DaysOld

### 4.4. Полнота и трассируемость
Для каждого задания **обязательно**:
- Лог-файл: `<PCName>_<JobName>_<timestamp>.log`
- XML-отчёт: `<PCName>_<JobName>_<timestamp>.xml`
- CSV-список файлов: `<PCName>_<JobName>_<timestamp>.csv`
- Сводный отчёт: `<PCName>_summary.xml`

## 5. Логирование, ошибки и наблюдаемость
### 5.1. Обязательное логирование
**Уровни:** `INFO`, `WARNING`, `ERROR`, `SUCCESS`, `DEBUG`

**Все ключевые этапы логгируются:**
- Старт скрипта и заданий
- Архивация (успех/ошибка)
- Верификация (результат сравнения)
- Копирование в сеть
- Ротация файлов
- Отправка почты

### 5.2. Обработка ошибок
**Обязательная конструкция:**
```powershell
try {
    # Критическая операция
}
catch {
    Write-Log "Ошибка: $($_.Exception.Message)" -Level ERROR
    throw
}
```

**Запрещён:** `SilentContinue`, `SilentlyContinue` для критических операций.

### 5.3. Безопасность логов
**ЗАПРЕЩЕНО** в логах:
- Пароли
- Токены
- Строки подключения
- Персональные данные

### 5.4. Уведомления
Email-уведомления:
- Сообщают агрегированный результат (Success/Error/Warning)
- Содержат ссылки на лог-файлы
- **Не дублируют** доменную логику
- Сбой отправки = `WARNING`, не `ERROR` бэкапа

## 6. Конфигурация и безопасность
### 6.1. XML-конфигурация (единый источник правды)
**Файл:** `Backup-Config-All.xml`

**Структура:**
```xml
<BackupConfig>
  <General>...</General>
  <Paths>...</Paths>
  <Recipients>...</Recipients>
  <Integrity>...</Integrity>
  <Jobs>...</Jobs>
</BackupConfig>
```

**Все пути, параметры и настройки — только в XML.**
Хардкод значений в скрипте **запрещён** (кроме констант по умолчанию).

### 6.2. Проверка целостности (SHA256)
**Обязательные проверки перед запуском:**
1. Хеш XML-конфигурации (`$XmlHash`)
2. Хеш RAR.exe (`RarExeHash` в конфиге)

**При несовпадении хеша — немедленная остановка скрипта.**

### 6.3. Минимизация внешних зависимостей
**Разрешено:**
- RAR.exe (указан в конфиге)

**Запрещено** без явного обоснования:
- Дополнительные конфигурационные файлы
- Сторонние модули
- Внешние API

### 6.4. Секреты и учётные данные
**ЗАПРЕЩЕНО** в скрипте и XML:
- Пароли
- Токены
- Ключи API

SMTP-аутентификация — через внешние механизмы или анонимный доступ.

### 6.5. Минимально необходимые права
Скрипт требует:
- Чтение: источники данных
- Запись: LocalDest, LogPathRoot, NetLogPath
- Выполнение: RAR.exe

## 7. Качество, тестирование и эволюция
### 7.1. Тестирование
**Обязательно:**
- Pester тесты в `app/tests/Backup.Tests.ps1`
- Покрытие: пустые директории, недоступное хранилище, ошибки RAR

### 7.2. Линтер
**Обязательная проверка:**
```powershell
powershell.exe -executionpolicy RemoteSigned -file .\app\PS_linter.ps1
```

### 7.3. Тестовый режим
**Всегда доступен:**
```powershell
powershell.exe -executionpolicy RemoteSigned -file .\Backup-ps2-v4.ps1 -testmode
```

Проверяет: архиватор, источники, права записи, SMTP.

### 7.4. Эволюция конституции
Все изменения должны:
1. Соответствовать текущей архитектуре (монолит, PS 2.0)
2. Не нарушать доменные инварианты
3. Фиксироваться с мотивацией

### 7.5. Ограничения для AI-агента
**AI-агенту ЗАПРЕЩЕНО:**
- Отключать логирование и обработку ошибок
- Нарушать порядок Unified Pipeline
- Добавлять PS 3.0+ возможности без совместимости
- Добавлять внешние зависимости без явного запроса
- Хардкодить пути и настройки (только через XML)

**При неоднозначности выбирать:** KISS, PS 2.0 совместимость, безопасность данных.
